1. Overview
The Revenue Router is the single point of capital inﬂow and allocation for the Sovereign 
M onad Ecosystem. It receives all external income from every source (M onadSpin provider 
revenue, M EV performance fees, Oracle API fees, Data licensing fees, trait-gated protocol 
fees, external framework licensing) and routes it to deﬁned sinks according to the ﬁnalized 
allocation policy.
Core Principle: Nothing leaks out by default. All routing is transparent, on-chain, and Dove-
observable.

3.4 Layer 4: Sinks (Destination Contracts)
Each sink is a simple, well-deﬁned contract that receives capital and emits events. Sinks are 
not part of the Router but are called by it.

5. Dove-Observable Hooks
The Revenue Router emits structured signals that the Dove Protocol observes to detect drift:
5.1 Tier 1 Signals (Silent Internal Flags)
Dove monitors these continuously:
• Allocation Accuracy: Is 100% of inﬂow allocated? (Drift if < 99.9%)
• Routing Success Rate: Are all sinks receiving capital? (Drift if < 99%)
• Inﬂow Consistency: Is revenue arriving as expected? (Drift if inﬂows drop > 20% week-
over-week)
• Sink Address Stability: Are sink addresses changing unexpectedly? (Drift if changes 
without governance event)
5.2 Tier 2 Signals (G overnance Proposals)
If Tier 1 signals detect drift, Dove generates governance proposals:
• "Allocation Policy Drift Detected" — If actual allocations deviate from policy
• "Routing Failure Rate Elevated" — If sink routing fails > 1%
• "Revenue Inﬂow Anom aly" — If inﬂows drop > 30% or spike > 50%
• "Sink Address U nauthorized Change" — If a sink address changes without DAO vote
5.3 Tier 3 Signals (Conﬁrm ation W ithdrawal)
If drift persists after Tier 2 signals are ignored:
• Dove presence dims across all products simultaneously
• On-chain alignment state shows "DRIFT DETECTED"
• All new inﬂows are held in escrow pending DAO resolution
6. Im plem entation Constraints
6.1 Security
• No custodial risk: Sinks are non-custodial, capital ﬂows directly from Router to 
destination
• Atom ic routing: All sinks receive capital in a single transaction (or all fail)
• Signature validation: All external inﬂows require provider signature
• Reentrancy protection: All sink calls use checks-eﬀects-interactions pattern
6.2 G as Eﬃciency
• Batch routing: M ultiple inﬂows can be routed in a single transaction
• M inim al storage: Only essential state stored on-chain (current policy, sink addresses, 
inﬂow history)
• Eﬃcient allocation m ath: Fixed-point arithmetic, no ﬂoating-point operations
6.3 Auditability
• Com plete event history: Every inﬂow, allocation, and routing is emitted as an event
• Im m utable records: All historical allocations stored in AllocationRecord structs
• Dove-readable state: All critical state queryable via view functions
